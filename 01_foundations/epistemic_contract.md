# The Epistemic Contract

CAP operates under an explicit epistemic contract. The contract states what the framework claims to be, what it does not claim to be, and what counts as evidence for or against its operation. Reading the contract before reading the framework is recommended; readers who skip it tend to either over-claim what CAP can do or under-use what it actually offers.

---

## What This Framework IS

CAP is an **observer-centric framework** for describing the organization of lived reality through:

- first-person experience as primitive
- continuity of the observer
- typed operator grammar
- transition cost
- observer budget
- telemetry gating
- risk-aware route engineering

It models not reality-in-itself, but the **navigable structure of experienced reality** as it becomes available to an embodied observer.

---

## Core Starting Condition

First-person experience is not derived. It is the primitive the framework begins from.

The framework does not attempt to explain *why* phenomenal experience exists. It assumes:

- experience is given first-personally
- continuity is pre-reflectively maintained
- all operator-routing remains embodied

This is not treated as a gap. It is the starting condition of the framework.

```text
Experience is not inferred by the model.
Experience is the condition under which the model becomes possible.
```

For the longer argument see [`primitive_experience.md`](./primitive_experience.md).

---

## What This Framework IS NOT

CAP is not:

- a complete theory of consciousness
- a solution to the hard problem
- a proven cosmology
- a claim that reality is literally an LLM
- a proof of many-worlds navigation
- a replacement for physics
- a deterministic fate system
- a divination system
- a symbolic oracle
- a license for magical thinking

Each of these has been a real failure mode in similar frameworks. CAP rules them out at the contract level so they cannot creep in through usage.

---

## LLM Metaphor Boundary

LLM terminology is used in CAP as an **engineering metaphor and modeling scaffold**.

It helps describe:
- constrained decoding
- context dependence
- route weighting
- latent transitions
- interpolation between typed states

But LLM behavior does not validate CAP ontology.

```text
LLM is a scaffold, not evidence.
```

The framework must avoid the loop:

```text
LLM metaphor describes reality
→ reality is interpreted through LLM
→ LLM behavior confirms the metaphor
```

This is invalid. Validation must come from:

- external observations
- operator success / failure
- telemetry signals
- behavioral outcomes
- reproducible route effects

If an LLM happens to produce CAP-shaped outputs, that is consistency with the framework, not proof of it. The framework is validated only by what happens when its operators are applied to real situations and the resulting telemetry is checked.

---

## Numeric Threshold Boundary

All numerical thresholds in the model — risk percentages, budget levels, telemetry gate cutoffs, leak level boundaries — are **provisional engineering defaults**.

They are **not**:
- ontological constants
- empirical proof of anything
- universal psychological laws
- argumentative authority

They **are**:
- runnable assumptions
- calibration targets
- sensitivity points
- test scaffolding

No threshold may be used as proof of the model. If changing a threshold changes the recommendation, the system must mark the decision as `threshold-sensitive / provisional`.

```text
Thresholds are knobs, not truths.
```

---

## Operational Validation Rule

A concept remains in the framework only if it improves at least one of:

- diagnosis
- operator selection
- risk estimation
- route success
- failure detection
- telemetry response
- reverse-trace accuracy

If it does not improve operation, it remains metaphor or narrative layer, not core model. This rule has already removed several concepts that initially looked promising. It is enforced strictly.

---

## Legacy Systems Boundary

I Ching, Tarot, hexagram-based systems, Medici Patience, astrology, and similar symbolic corpora may be used in CAP only as:

- **historical analogies** — a hexagram or card sequence that matches the current route configuration can be cited as a historical data point
- **symbolic frontends** — entry-point interfaces for users who find direct CAP syntax intimidating, provided the symbolic input is translated into the operator grammar before analysis
- **translation layers** — a legacy system's vocabulary can be mapped to CAP node states as a glossary, as a user-facing aid

They cannot:

- replace operator grammar as the diagnostic engine
- determine node states without telemetry grounding
- gate operator selection via symbolic readings
- compress state space to a fixed historical grid (64 hexagrams, 78 cards, etc.)

```text
Legacy systems compress reality.
COM parses it.
LLM interpolates it.
Telemetry limits it.
Risk throttling protects it.
```

The error is not using legacy systems. The error is confusing frontend with engine.

---

## The Strong Claim

The contract has a single strong consequence:

> **A recommendation that exceeds observer budget is structurally false for that observer at that time.**

This is not a soft claim about feasibility. It is a structural claim about validity. A CAP analysis that produces an operator recommendation without checking the observer's budget has produced an opinion in CAP syntax, not a CAP recommendation.

The contract holds the framework to this claim. Removing budget enforcement removes the framework's identity.
